# Editorial Expansion Guide

## Current State

MaestraAI currently supports **Richmond LP** as the primary editorial integration. The architecture is designed to support multiple editorials with minimal code changes.

## Architecture Overview

### Database Tables (Multi-Editorial Ready)

The following tables are **editorial-agnostic** and ready for expansion:

1. **`teachers`** - Has `editorial` field (text) to identify which editorial the teacher uses
2. **`richmond_credentials`** - Stores encrypted login credentials (editorial-specific)
3. **`richmond_sync_log`** - Tracks sync operations (editorial-specific)
4. **`richmond_assignments`** - Stores assignment data (editorial-specific)
5. **`richmond_scores`** - Stores student performance data (editorial-specific)

### Dynamic UI Components

All hardcoded "Richmond" references have been replaced with dynamic lookups:

- **Dashboard card**: Shows editorial name from `teacher.editorial`
- **Richmond dashboard page**: Title dynamically displays editorial name
- **Configuration page**: API key section shows editorial name

### Editorial Display Names

## How to Add a New Editorial (e.g., Macmillan)

### Step 1: Database Setup

Create new tables following the Richmond pattern (4 tables per editorial).

**Pattern**: `{editorial}_credentials`, `{editorial}_sync_log`, `{editorial}_assignments`, `{editorial}_scores`

### Step 2: API Client Implementation

Create `/lib/macmillan/api-client.ts`:

```typescript
// Follow the same pattern as /lib/richmond/api-client.ts
// Implement authentication, session management, data fetching
```

### Step 3: Sync Service

Create `/lib/macmillan/sync-service.ts`:

```typescript
// Follow the same pattern as /lib/richmond/sync-service.ts
// Implement student matching, assignment syncing
```

### Step 4: API Routes

Create API endpoints:

- `/app/api/macmillan/sync/route.ts` - Manual sync trigger
- `/app/api/macmillan/ingest/route.ts` - Chrome extension webhook
- `/app/api/macmillan/upload-xlsx/route.ts` - CSV import (optional)

### Step 5: Chrome Extension

Update extension to detect Macmillan platform:

```javascript
// extension/content.js
if (window.location.hostname.includes('macmillan')) {
  // Inject Macmillan scraper
}
```

### Step 6: Update UI Constants

Add to `/app/(main)/dashboard/richmond/page.tsx`:

```typescript

  richmond: 'Richmond LP',
  macmillan: 'Macmillan Education', // ADD THIS
  pearson: 'Pearson',
}
```

### Step 7: Test

3. Test Chrome extension scraping

## Key Architecture Decisions

1. **Separate tables per editorial** - Avoids polymorphic relationships, cleaner queries
2. **Editorial field in teachers table** - Single source of truth for which editorial a teacher uses
3. **Dynamic UI rendering** - No hardcoded "Richmond" in components
4. **Reusable sync patterns** - Each editorial follows same sync flow
5. **RLS at table level** - Each editorial has own security policies

## Migration Checklist

When adding a new editorial:

- [ ] Create database tables (`{editorial}_credentials`, `{editorial}_sync_log`, etc.)
- [ ] Add RLS policies
